## 1. Architecture Identification

The system remains a **Node.js/Express monolith with layered architecture** and a single composition root (`createControllers` / `createApp` in [app.js:56-172](../app.js#L56-L172)). The actual topology, post-AUDIT-3 refactor wave, is:

```text
routes/*               (HTTP transport â€” Express routers + role guards)
   â†“
controllers/*          (HTTP I/O adapters â€” payload normalisation, response envelopes)
   â†“
services/*             (use cases â€” orchestration, ServiceError contracts)
   â†“
repositories/*         (persistence â€” Prisma queries grouped by aggregate)
   â†“
prisma/client.js       (singleton Prisma + MariaDB adapter)
domain/spanish/*       (rule + LLM checkers, semantic orchestrator)
entities/*             (DTOs: User, EntryDTO, DatasetDTO, DatasetListItemDTO)
contracts/dto-mappers  (boundary normalization)
utils/*                (XML, validators, LLM HTTP, temp storage, alerts, api-error-payload)
constants/*            (status enums, roles, validation codes, paths)
middlewares/*          (auth, multer upload, request logging)
```

The intent is a **layered architecture with explicit dependency injection at the composition root**. It is not hexagonal (no formal ports/adapters; Prisma is referenced directly by repositories) and not microservices (single process, single DB).

### 1.2 Coupling

* **`spanish-service` dual export**: [spanish-service.js:857-863](../domain/spanish/spanish-service.js#L857-L863) constructs `const defaultSpanishService = createSpanishService();` at load time and then exports `{ ...defaultSpanishService, createSpanishService }`, immediately followed by `module.exports.createSpanishService = createSpanishService;`. Two simultaneous shapes (factory + flattened default singleton) and a redundant re-assignment. Every other domain/service module exposes only the factory.
* **`spanish-service.save` legacy branch**: the method still accepts a positional `(rdfId, sentence, rejectionReason)` triple and returns a no-op `{ ok:true, rdfId, sentence, rejectionReason }` ([spanish-service.js:102-109](../domain/spanish/spanish-service.js#L102-L109)). No caller exercises this path: `annotations-service.persistAnnotation` always invokes the object form. Dead, but reachable from tests and external scripts.
* **Section finalization split between two services**: `annotations-service.finalizeSectionIfRequested` ([annotations-service.js:182-202](../services/annotations-service.js#L182-L202)) invokes `sectionAssignmentService.completeAssignmentIfSectionDone` AND, conditionally on `shouldFinalizeSection`, `datasetsRepository.markSectionAsAnnotated`. The two updates are not transactional; the first can succeed and the second never run if `shouldFinalizeSection === false` while annotated rows complete the section â€” counters drift.
* **Two section selection algorithms over the same aggregate**: `sectionAssignmentService.requestSection` picks by complexity / availability ([section-assignment-service.js:52-131](../services/section-assignment-service.js#L52-L131)); `continueDatasetService.continueDataset` picks `maxSectionIndex + 1` sequentially ([continue-dataset-service.js:111-122](../services/continue-dataset-service.js#L111-L122)) and then delegates the write to `sectionAssignmentService.assignSection`. Write is unified; the *selection policy* is not.
* **Implicit dual graphs through DI fallbacks**: every service still has `dep || createXRepository()` defaults â€” [datasets-service.js:45-52](../services/datasets-service.js#L45-L52), [annotations-service.js:32-38](../services/annotations-service.js#L32-L38), [continue-dataset-service.js:28-43](../services/continue-dataset-service.js#L28-L43), [section-assignment-service.js:40-45](../services/section-assignment-service.js#L40-L45), [reviews-service.js:51-56](../services/reviews-service.js#L51-L56), [admin-service.js:16-20](../services/admin-service.js#L16-L20). The composition root injects everything in production, but any direct `require` builds a second graph against the same global Prisma singleton.
* **Two parallel HTTP logging pipelines**: `pinoHttp({ autoLogging: false })` ([app.js:117-118](../app.js#L117-L118)) injects `request.log` but autoLogging is disabled and no application code consumes the logger; meanwhile [`request-log-middleware.js`](../middlewares/request-log-middleware.js) appends to `logs/YYYY-MM-DD-HH.txt` / `logs/YYYY-MM-DD-error.txt` through a singleton write queue. One of the two stacks is dead weight.
* **`entities/dataset.DatasetListItemDTO` vs `contracts/dto-mappers.mapDatasetListDTO`**: both shape the dataset list payload, with separate normalisation helpers (`normalizeName`, `normalizeColorClass`, `normalizePercent`, `toIntegerNormalized`) and slightly different output shapes. Controllers only call `mapDatasetListDTO`; `DatasetListItemDTO.fromSource(...)` is unused outside tests.
* **`users-controller` outside the unified envelope**: it imports neither `respondWithApiError` nor `resolveSessionUserId`, sets no `response.locals.serverErrorReason`, and returns `{ message }`/`{ title, message }` payloads ([users-controller.js:32-51, 73-99, 113-134](../controllers/users-controller.js#L32-L134)). The frontend therefore has to handle two error shapes depending on the endpoint family.

## 4. Executive summary

* The architecture has continued to mature since AUDIT-3: the `business/` folder is gone, controllers and the domain layer have their own homes, the LLM HTTP surface is factored, the API error envelope is unified across five of six controllers, and the BLOB/relational duality on `Dataset.content` is resolved. The session store is now Prisma-backed and the legacy `mysql` driver is gone.
* **Three structural debts still dominate** and define the next refactor wave:
  1. **`domain/spanish/spanish-service.js` remains the largest non-cohesive module** (findings 1, 2, 3, 13). It mixes orchestration, an entity/predicate lexicon, a coverage heuristic, alert post-processing and persistence dispatch, retains a dead legacy `save` branch, and is the only module in the codebase that exports both a flattened default singleton AND a factory.
  2. **Section-assignment lifecycle is still split between two services with two selection policies** (#5), and `annotations-service.saveSentences` still performs four non-transactional writes across three aggregates (#4). Combined with the denormalised section counters on `Dataset` (#17), a partial failure mid-`saveSentences` produces silent drift between `section_assignments.status` and `Dataset.sectionsPending/Completed/InReview`.
  3. **Contract drift in the boundary layer**: `dto-mappers.js` carries the entire alias compatibility burden (#16), `DatasetListItemDTO` duplicates `mapDatasetListDTO` (#9), `User`'s persistence vs session shapes disagree on the id key (#12), and `nick` survives in outbound DTOs after the underlying surface moved to `email` (#22).
* **Cheap clean-up wins** worth bundling in one pass: align `users-controller` with the unified envelope (#7, #8), pick one logging stack (#11, #24), strip placeholder JSDoc (#14), move `typescript` to devDependencies (#20), and converge the middlewares to factories (#18). None of these is architectural but together they remove the bulk of the remaining stylistic noise.
* **No new dependency-injection or secrets issues** were detected: the composition root injects all controllers explicitly in `app.js`, secrets are env-driven, passwords go through scrypt + `timingSafeEqual`, and the single error handler funnels every unhandled error through `respondWithApiError` or the global handler. The DI fallback `dep || createXRepository()` (#6) is now the only structural risk to "one graph per process".
