# Legacy Architecture Audit — AUDITORY-6

**Date:** 2026-05-19
**Auditor:** Claude Opus 4.7
**Scope:** application source code of `lanbench` (excluding `node_modules/`, `public/`, `views/`, `logs/`, `uploads/`, `tmp/`, `test_datasets/`, `front-mocks/`, `tests/` — tests are consulted as consumers when judging unused exports).
**Type:** *Legacy* snapshot. This audit introduces **no new findings**. Its purpose is to (a) re-evaluate every anomaly catalogued in [AUDITORY-1](AUDITORY-1.md), [AUDITORY-2](AUDITORY-2.md), [AUDITORY-3](AUDITORY-3.md), [AUDITORY-4](AUDITORY-4.md) and [AUDITORY-5-DUPLICATIONS](AUDITORY-5-DUPLICATIONS.md) against the current source tree, and (b) re-scope each surviving anomaly against the architecture and user stories that the project has actually committed to ([documentation/USER-STORIES.md](../documentation/USER-STORIES.md), [documentation/TECHNICAL-DESIGN.md](../documentation/TECHNICAL-DESIGN.md), [prisma/schema.prisma](../prisma/schema.prisma)).

The earlier audits described a codebase whose layering and naming have since changed (`business/` → `controllers/` + `domain/spanish/`, `usuario.js` → `user.js`, hand-rolled session store → `PrismaSessionStore`, BLOB-on-Dataset → persisted graph, `nick` → `email`). This audit re-uses the original finding IDs as anchors but **rewrites every Scope and Description against the current file/function names** so the row is actionable today.

**Per-finding methodology (every row was checked this way):**

1. Re-read the *current* file(s) at the line range pointed by the originating audit. If lines shifted, the row was updated to today's coordinates.
2. Confirmed whether the symptom is still observable. If not, the row was promoted to the §0 baseline.
3. Cross-checked the scope against [USER-STORIES.md](../documentation/USER-STORIES.md) and [TECHNICAL-DESIGN.md](../documentation/TECHNICAL-DESIGN.md): when the documented design *prescribes* the behaviour the audit objected to (e.g. denormalised section counters in §3.5, hand-rolled admin export XML in §7.3), the row was rewritten to target only the residual sub-symptom (atomicity, not denormalisation; shared escaper, not two emitters).

## 1. Re-scoping notes (designed-in behaviours)

Three originating findings asked for refactors that are now in tension with the project's documented design. The audit rows that survive in §2 are scoped to the residual sub-symptom, not to the documented decision.

- **Denormalised section counters (AUDIT-4 #17).** [TECHNICAL-DESIGN.md §3.5](../documentation/TECHNICAL-DESIGN.md#35-progress-counters) explicitly chooses to persist `Dataset.sectionsCompleted` / `sectionsInReview` / `sectionsPending` as application-maintained counters — "*they are not derived on the fly*". The audit's earlier proposal to recompute on demand contradicts that decision. The row in §2 is rewritten to target only the **atomicity** of the chain `completeAssignmentIfSectionDone` → `markSectionAsAnnotated` in `finalizeSectionIfRequested`; the counters themselves stay denormalised.
- **Two XML emitters (AUDIT-3 #7 / AUDIT-4 #10 / AUDIT-5 §1.7).** [TECHNICAL-DESIGN.md §7.3](../documentation/TECHNICAL-DESIGN.md#73-export-endpoint) treats `admin-service.buildExportXml` as a separate emitter for the admin progress export (US-20), distinct from `utils/dataset-xml.js#buildDatasetXml` used by the user-facing downloads `GET /api/datasets/:id/download` and `/download/annotated` (US-29 / US-30). The split between layers is intentional; the row in §2 is rewritten to target only the **shared low-level machinery** (open-tag helper, attribute escaping) that should live in `utils/xml-format.js` and be consumed by both.
- **Section selection algorithm (AUDIT-4 #5).** [TECHNICAL-DESIGN.md §3.4](../documentation/TECHNICAL-DESIGN.md#34-section-assignment-algorithm) canonicalises `nextIndex = maxSectionIndex + 1` (sequential) as **the** selection rule used by `continueDataset`. The complexity-filtered alternative in `section-assignment-service.requestSection` is not the canonical algorithm and is **no longer reached from any production code path** (only tests call it). The row in §2 is therefore rewritten as a *dead-code* finding, not a *duplicated-logic* finding.

- **This audit is a legacy re-evaluation. No finding type is new.** AUDITORY-1 and most of AUDITORY-2 are fully closed: the dual data-access path, the orphan relational schema for entries/triples/lexes, the bilingual entity, the `MAX(id)+1` IDs, `/tmp` coupling, `/fake-login`, plaintext passwords, hardcoded credentials, multiple express.static mounts, and the AUDITORY-4 tool-generated JSDoc placeholders are all gone. - **Three structural debts dominate the unresolved set**, unchanged in shape since AUDITORY-3:
  1. **`domain/spanish/spanish-service.js`** remains the largest non-cohesive module (AUDIT-1 #10 / AUDIT-3 #1 / AUDIT-4 #1 / AUDIT-5 §4.7) and carries two sibling debts: the hardcoded NLP lexicon (AUDIT-4 #13 / AUDIT-5 §4.6) and the 9 test-only helpers exported by `ollama-spanish-checker.js` (AUDIT-5 §4.5).
  2. **The section-close chain in `annotations-service.saveSentences` is still non-transactional** (AUDIT-3 #2 / AUDIT-4 #4 / AUDIT-4 #17). Per [TECHNICAL-DESIGN.md §3.5](../documentation/TECHNICAL-DESIGN.md#35-progress-counters) the section counters on `Dataset` are intentionally denormalised — the open issue is that `completeAssignmentIfSectionDone` and `markSectionAsAnnotated` are not wrapped in a single transaction, so the counters can drift from `section_assignments.status` if the chain partially fails.
