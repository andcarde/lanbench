## 1. Architecture Identification

The system is a **Node.js/Express monolith with layered architecture**, with partial MVC intent but without strict adherence. The observed topology is:

```text
routes/*            (HTTP transport layer â€” Express routers)
   â†“
business/*          (application layer â€” controllers + orchestration)
   â†“
prisma/client.js    (persistence layer â€” ORM)
entities/*          (anemic DTOs / VOs)
utils/*             (helpers: XML, validation, Ollama)
```

No signs of **hexagonal architecture** are observed (there are no explicit ports/adapters, and infrastructure dependencies â€”Prisma, `/tmp` filesystem, Ollama HTTPâ€” are consumed directly from the application layer), nor of **microservices** (single Express process, single `package.json`, single database).

### 1.1 Module cohesion

| Module          | Cohesion        | Observation                                                                                                                                                    |
| --------------- | --------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `business/*`    | **Medium-low**  | Controllers mix HTTP payload validation, orchestration, Prisma queries, DTO mapping, and response formatting. A single handler accumulates 4 responsibilities. |
| `utils/*`       | **Medium**      | Grouped by purpose, but `xml-utils.js` is a useless shim and `xml-writer.js` diverges in style (callback-based).                                               |
| `routes/*`      | **Medium-high** | Thin routers, with clear separation between pages (HTML) and API (`/api/...`).                                                                                 |
| `entities/*`    | **Low**         | Three different export conventions (`User` class, `Dataset` with two classes in an object, `Entry` standalone class without `'use strict'`).                   |
| `middlewares/*` | **High**        | Each middleware has a single well-defined responsibility.                                                                                                      |

### 1.2 Coupling

* **Controllers â†’ direct Prisma usage** (no repository layer). Any ORM change affects N controllers.
* **Controllers â†’ `/tmp` filesystem** (coupling to OS infrastructure; on Windows it is created at the disk root).
* **Routes â†’ controllers with two construction styles** simultaneously: factory with DI (`createDatasetsApiRouter({ datasetsController })`) *and* default singleton instantiated as a side effect of `require`. **Unjustified duality.**
* **Log middleware â†” controllers**: implicit contract through `response.locals.serverErrorReason`; not all controllers set it.
* **Session store â†” config**: MySQL credentials and session secret hardcoded in `config.js` and `routes/session.js`.

| Principle                       | Status                      | Deviation                                                                                                              |
| ------------------------------- | --------------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| **SRP** (Single Responsibility) | âš ď¸Ź violated                 | HTTP handlers mix validation + Prisma + mapping + response formatting.                                                 |
| **DIP** (Dependency Inversion)  | âš ď¸Ź partial                  | The factory accepts injected dependencies, but coexists with a default singleton â†’ two simultaneous dependency graphs. |
| **OCP / LSP**                   | âś… not particularly relevant | â€”                                                                                                                      |
| **Separation of concerns**      | âš ď¸Ź partial                  | Absence of `repositories/` layer and `services/` layer separated from `controllers/`.                                  |
| **Modularity**                  | âś… reasonable                | Well-delimited directories, clean relative imports.                                                                    |
| **Secrets in code**             | âťŚ                           | MySQL root/'' and session secret committed in the repository.                                                          |
| **Observability**               | âš ď¸Ź                          | Silent `catch (_error)` blocks in several handlers without logging or propagating the reason.                          |

## 3. Justification for model assignment

Applied criterion: **lowest power with success rate â‰Ą 95%**, considering problem type+complexity and the model's reasoning+context capabilities.

* **Claude Haiku Thinking** â†’ assigned to mechanical and local tasks: removal of unused imports, simple dead code, renames, consolidation of export conventions, small schema adjustments. Reduced context volume (1-2 files), straightforward reasoning, low risk.
* **Claude Sonnet 4.6 Medium** â†’ assigned to tasks with cross-cutting but bounded implications: moving secrets to environment variables, introducing `bcrypt`, unifying the error handler, replacing `/tmp` with `os.tmpdir()`, implementing annotation persistence. Requires coordination across 2-5 files and API decisions.
* **Claude Sonnet 4.6 High** â†’ assigned to architectural refactors: unifying controller construction patterns (removing dual default singleton), introducing `repositories/` layer and extracting `services/`, and deciding the entry persistence strategy (BLOB vs relational). These tasks affect domain structure and require structural reasoning over the complete graph.
* **Claude Opus 4.7 Medium / High** and **ChatGPT 5.4** â†’ **not assigned**. No problem requiring stronger reasoning was identified: there are no complex algorithms, ambiguous intent inference, or context volumes exceeding Sonnet 4.6 capabilities. Escalating to these models would be overengineering.

## 4. Executive summary

* The base architecture is **appropriate for the project size** (layered Express + Prisma monolith) and has absorbed the findings from AUDIT-1 well.
* The **two remaining major structural debts** are:

  1. **Dual controller instantiation** (DI factory + default singleton coexisting). This is the root of style inconsistency in `business/` and `routes/*-api.js`.
  2. **Orphan relational entry schema** (`Entry`/`Tripleset`/`Triple`/`Lex`/`Dbpedialink`/`Link` without readers or writers).
* The unresolved **security priorities** from AUDIT-1 are: plain-text passwords, hardcoded secrets+credentials, and coupling to `/tmp`.
* The rest are minor smells (redundancy, naming, dead dependencies).
