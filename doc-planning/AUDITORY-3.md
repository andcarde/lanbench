# Architecture Audit — AUDITORY-3

**Date:** 2026-05-18
**Auditor:** Claude Opus 4.7
**Scope:** application source code of `lanbench` (excluding `node_modules/`, `public/`, `views/`, `logs/`, `tmp/`, `tests/`).
**Previous baseline:** [`AUDITORY-2.md`](AUDITORY-2.md). Several findings from AUDITORY-2 are now mitigated and are not repeated here:
- Persistent layered split is materialized: `routes/ → business/ (controllers) → services/ → repositories/ → prisma/`.
- Passwords are hashed with scrypt (`services/password-hasher.js`) and verified via `timingSafeEqual`. Legacy plaintext rows trigger an opportunistic upgrade.
- Secrets and DB credentials moved to `.env`/`process.env` (`config.js`).
- `User.email` is `@unique` in `prisma/schema.prisma`.
- A single Express error handler (`createErrorHandler` in [app.js:178](app.js#L178)) replaces the three per-status handlers.
- `express.static(public)` is mounted once.
- `download_datasets.js` was removed.

## 1. Architecture Identification

The system is a **Node.js/Express monolith with a clean layered architecture** and a clearly identifiable composition root (`createControllers` / `createApp` in [app.js](app.js)). The actual topology is now:

```text
routes/*               (HTTP transport — Express routers, role guards)
   ↓
business/*             (controllers — HTTP I/O adapters)
   ↓
services/*             (use cases — orchestration, ServiceError contracts)
   ↓
repositories/*         (persistence — Prisma queries grouped by aggregate)
   ↓
prisma/client.js       (singleton Prisma + MariaDB adapter)
entities/*             (anaemic DTOs: User, EntryDTO, DatasetListItemDTO)
contracts/dto-mappers  (boundary normalization)
utils/*                (XML, validators, LLM clients, temp storage, alerts)
constants/*            (status enums, roles, validation codes)
middlewares/*          (auth, multer upload, request logging)
```

The intent is a **layered architecture with explicit dependency injection at the composition root**. It is not hexagonal (no formal ports/adapters; Prisma is referenced directly by repositories) and not microservices (single process, single DB).

### 1.1 Module cohesion

| Module           | Cohesion        | Observation                                                                                                                                                                                                                                              |
| ---------------- | --------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `routes/*`       | **High**        | Thin routers, role/auth guards applied at the boundary. Two style families remain (default singleton vs factory) — see below.                                                                                                                            |
| `business/*`     | **Low**         | Mixes HTTP controllers, domain rule-checkers (`rule-checker`, `ollama-spanish-checker`, `spanish-service`, `spanish-draft-generator`), dead semantic helpers (`triple-coverage-checker`, `diversity-checker`). The folder no longer represents one role. |
| `services/*`     | **Medium-high** | Each service owns one use case (datasets/annotations/reviews/admin/users). Side concerns leak: `annotations-service` reaches into datasets/section-assignment repositories and active-session mutations within `saveSentences`.                           |
| `repositories/*` | **High**        | Each file maps to an aggregate (datasets, users, reviews, annotations, evaluation-criteria, section-assignments, active-sessions). Prisma is the sole dependency.                                                                                        |
| `entities/*`     | **Low**         | Three conventions persist: `User` (full class with `fromSession`/`toSession`), `dataset.js` exports `{ DatasetDTO, DatasetListItemDTO }` (mixed-style classes), `entry.js` exports `EntryDTO` as a bare data holder. No shared base or convention.       |
| `utils/*`        | **Medium-low**  | Useful XML helpers coexist with duplicates (`xml-writer` vs `dataset-xml`) and parallel LLM client copies (`ollama-client` vs `groq-client`) plus a dispatcher (`llm-client`).                                                                            |
| `middlewares/*`  | **High**        | Auth, upload, logging — each one well-scoped.                                                                                                                                                                                                            |
| `contracts/*`    | **Medium**      | `dto-mappers.js` centralizes outbound shaping, but accepts ~8 alias keys per field (`id` / `datasetId`, `englishSentences` / `sourceSentences`, …), hiding upstream contract inconsistencies instead of fixing them at the source.                       |

### 1.2 Coupling

* **Services ↔ default repositories**: factories like `createDatasetsService({...})` fall back to `createDatasetsRepository()` when a dependency is missing ([datasets-service.js:51-60](services/datasets-service.js#L51-L60)). The composition root injects them explicitly, but tests and ad-hoc calls silently create second graphs talking to the same global Prisma singleton.
* **`continueDatasetService` ↔ `sectionAssignmentService`**: both manage section assignments lifecycle (`expireStaleAssignments`, `findActiveAssignment`, `createAssignment`, `completeAssignmentIfSectionDone`). `continueDataset` creates assignments inline ([continue-dataset-service.js:115](services/continue-dataset-service.js#L115)) bypassing `SectionAssignmentService.requestSection`. Two parallel control flows over the same aggregate.
* **`annotations-service.saveSentences`** ([annotations-service.js:77](services/annotations-service.js#L77)) directly orchestrates spanish-service persistence, then mutates `sectionAssignmentService`, then `continueDatasetService.advanceSession`, then `datasetsRepository.markSectionAsAnnotated`, then re-calls `sectionAssignmentsRepository.updateAssignmentStatus`. Four cross-aggregate writes in one method.
* **`spanish-service.js`** (864 lines): mixes ruleBasedChecker + semanticChecker orchestration, hardcoded `ENTITY_ALIASES` / `PREDICATE_RELATION_PATTERNS` / `COMPLETE_SENTENCE_MARKERS`, alert merging, false-positive suppression, persistence delegation. Three responsibilities collapsed.
* **`routes/session.js`** ↔ `mysql` package: opens an independent `express-mysql-session` MySQL pool at module load using `mysql` driver ([session.js:9-18](routes/session.js#L9-L18)), while `prisma/client.js` connects to the same DB through `@prisma/adapter-mariadb`. Two drivers, two pools, two connection configs for one database.
* **Two construction styles** still coexist in business: `createDatasetsController({...})` (factory + DI) and `business/spanish-service.js` (exports both `defaultSpanishService` spread AND `createSpanishService` factory, then re-assigns one onto `module.exports.createSpanishService` for safety — [spanish-service.js:857-863](business/spanish-service.js#L857-L863)). The dual export pattern from AUDIT-2 is gone for controllers but persists in `spanish-service`.

### 1.3 Comparison with industry standards (SOLID + others)

| Principle                       | Status         | Deviation                                                                                                                                                                                                                                                  |
| ------------------------------- | -------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **SRP** (Single Responsibility) | ⚠️ violated    | `business/spanish-service.js` (864 lines, ≥4 responsibilities); `annotations-service.saveSentences` (4 cross-aggregate writes); `business/datasets-controller.js` (mix of session resolution, error mapping, temp-file cleanup, dataset routing).          |
| **OCP** (Open/Closed)           | ⚠️ partial     | `rule-checker.js` hardcodes Spanish error patterns (`/\bago\b/`, `/\buna\s+l[aá]piz\b/`) instead of being rule-driven; `spanish-service.js` embeds entity aliases as a module-level Map; adding a new rule requires editing the checker.                   |
| **LSP** (Liskov Substitution)   | ✅              | No relevant inheritance hierarchies.                                                                                                                                                                                                                       |
| **ISP** (Interface Segregation) | ⚠️             | Services interrogate the optional shape of injected repositories with `typeof repo.X === 'function'` ([annotations-service.js:166](services/annotations-service.js#L166-L168), [datasets-service.js:82,102,412](services/datasets-service.js#L82)). Implicit fat interfaces. |
| **DIP** (Dependency Inversion)  | ⚠️ partial     | Services accept injected deps but instantiate concrete fallbacks (`||  createXRepository()`); there is no abstract `Repository` interface. Two concurrent dependency graphs are possible.                                                                  |
| **Separation of concerns**      | ✅ mostly       | Clear `routes → controllers → services → repositories` layering is achieved. The remaining seams are inside `business/*` (controllers + domain mixed) and inside services (cross-aggregate orchestration).                                                 |
| **Don't Repeat Yourself**       | ⚠️             | Several true duplicates: LLM HTTP helpers, XML builders, coverage logic, error envelopes, session-user resolvers (see §2).                                                                                                                                 |
| **Modularity**                  | ✅              | Folder boundaries are clean; relative imports are well-formed.                                                                                                                                                                                             |
| **Express error handling**      | ✅              | Single centralized `errorHandler` in [app.js:182](app.js#L182).                                                                                                                                                                                            |
| **Observability**               | ⚠️             | `response.locals.serverErrorReason` is still the implicit contract between controllers and `request-log-middleware`; some controllers set it (datasets, admin), others do not (reviews, users, annotations).                                               |
| **API contract consistency**    | ⚠️             | Three different JSON error envelopes coexist (see §2 #6); endpoint `POST /create-session` is non-RESTful; `POST /api/datasets/:id/permissions` silently ignores the role payload.                                                                          |

## 3. Justification for model assignment

Applied criterion: **lowest power among the allowed models with success rate ≥ 90% on the task type and complexity**. The available pool is restricted to `Claude Sonnet 4.6 High`, `Claude Opus 4.7 High`, and `Claude Opus 4.7 Very High`.

* **Claude Sonnet 4.6 High** → assigned to *Minimal* and *Low* complexity findings: mechanical clean-ups, file consolidation with limited reasoning surface, route renames, JSDoc strip-down, single-file refactors, entity convention unification. These are 2–4 files, no architectural decision required.
* **Claude Opus 4.7 High** → assigned to *Medium* complexity findings that require cross-file reasoning but not domain reshaping: error-envelope unification (12), shared LLM HTTP layer (6), coverage logic consolidation (8), wiring or removing dead semantic modules (9), choosing the session store strategy (13), enforcing DI everywhere (24), splitting `business/` (17), reconciling dataset BLOB vs entries (16), and fixing the dead `completeAssignmentIfSectionDone` path (5).
* **Claude Opus 4.7 Very High** → assigned to *High* complexity findings that require holistic reasoning over the domain model and aggregate boundaries: decomposing `spanish-service.js` (1), redesigning `saveSentences` cross-aggregate orchestration (2), unifying the two assignment lifecycles (3), and aligning the `Annotation` schema with the actual persistence shape (4). Each touches multiple aggregates simultaneously and requires choosing a target invariant before refactoring.

No finding warranted further escalation. Sonnet 4.6 Medium and Opus 4.7 Medium are excluded by the brief; Sonnet 4.6 High serves as the floor.

## 4. Executive summary

* The architecture has matured substantially since AUDIT-2: there is now a real `services/` + `repositories/` split, a single composition root, scrypt-hashed passwords, environment-driven configuration, and a single error handler. The dual-singleton controller construction is gone for controllers.
* **Three structural debts remain dominant** and define the next refactor wave:
  1. **`business/spanish-service.js` and the surrounding domain layer** (problems 1, 8, 9, 17, 18) — the file is too large, duplicates coverage logic with an unused helper, ships dead diversity/draft modules, and lives in a folder whose role is unclear.
  2. **Section-assignment lifecycle is duplicated** between `continue-dataset-service` and `section-assignment-service` (problems 3, 5), and `annotations-service.saveSentences` orchestrates four cross-aggregate writes (problem 2). One owner per aggregate is needed.
  3. **`Annotation` schema vs persistence collapse** (problem 4): the table is keyed `[entryId, datasetId]` but the code joins N sentences into a single row. The schema and the code disagree on the intended cardinality.
* **Library and contract clean-up** is the cheapest layer of fixes: unify the three error envelopes (12), share the LLM HTTP plumbing (6), pick one XML builder (7, 26), and pick one DB driver (13, 23). These are mostly Sonnet 4.6 High tasks and would visibly reduce noise across the codebase.
* **Two open security/UX contracts** worth flagging even outside the §2 priority list: `POST /api/datasets/:id/permissions` silently downgrades the requested role to "annotator" (14), and the "nick" surface is actually email-based (15). Both are small but mislead callers.
