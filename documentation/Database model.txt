# Trabajo de Fin de Master (TFM)

## Modelo Entidad-Relación

En el presente documento se define el modelo de la base de datos de la aplicación, orientado al almacenamiento de datasets, sus secciones internas y los permisos de acceso de los usuarios sobre dichos datasets.

# Lista de entidades

Se identifican las siguientes entidades principales dentro del sistema:

- **Section**: representa una sección o bloque concreto perteneciente a un dataset.
- **Dataset**: representa un conjunto de datos almacenado en el sistema, incluyendo métricas de progreso y metadatos de presentación.
- **User**: representa a un usuario registrado en la aplicación.
- **Permits**: representa la relación de permisos entre usuarios y datasets.
- **entry**: representa una entrada individual del benchmark WebNLG, identificada por su `eid` y clasificada por categoría.
- **tripleset**: agrupa los triples RDF de una entrada, diferenciando entre el conjunto original y el modificado.
- **triple**: almacena cada triple RDF individual (sujeto, predicado, objeto) dentro de un tripleset.
- **lex**: almacena las frases de referencia asociadas a una entrada, con soporte multilingüe agrupado por identificador de frase (`lid`).
- **dbpedialink**: almacena los enlaces entre entidades de DBpedia en distintos idiomas (relaciones `sameAs` entre versiones lingüísticas).
- **link**: almacena enlaces libres entre entidades que no figuran en DBpedia.

# Modelo de la base de datos

# Section
Section {
    idSection : INT [NOT NULL] [PRIMARY KEY]
    idDataset : INT [NOT NULL] [FOREIGN KEY -> Dataset.idDataset]
    block : INT [NOT NULL]
}

**Descripción**:  
La entidad `Section` almacena las distintas secciones en las que puede dividirse un dataset. Cada sección pertenece obligatoriamente a un único dataset y queda identificada por su `idSection`. El atributo `block` permite representar el bloque o fragmento asociado dentro del dataset.

# Dataset
Dataset {
    idDataset            : INT [NOT NULL] [PRIMARY KEY]
    name                 : VARCHAR(128) [NOT NULL]
    entries              : INT [NOT NULL]
    content              : BLOB [NOT NULL]
    languages            : TEXT [NULL]
    completedPercent     : TINYINT UNSIGNED [NOT NULL] [DEFAULT 0]
    withoutReviewPercent : TINYINT UNSIGNED [NOT NULL] [DEFAULT 0]
    remainPercent        : TINYINT UNSIGNED [NOT NULL] [DEFAULT 100]
    colorClass           : VARCHAR(64) [NOT NULL] [DEFAULT 'dataset-purple']
    createdAt            : DATETIME [NOT NULL] [DEFAULT CURRENT_TIMESTAMP]
    updatedAt            : DATETIME [NOT NULL] [DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP]
}

**Descripción**:
La entidad `Dataset` representa un conjunto de datos almacenado en la aplicación. Incluye un identificador único (`idDataset`), un nombre descriptivo (`name`), el número de entradas asociadas (`entries`) y el contenido bruto del dataset (`content`). El campo `languages` almacena los idiomas presentes en el dataset. Los campos `completedPercent`, `withoutReviewPercent` y `remainPercent` registran el estado de progreso del trabajo de anotación. El campo `colorClass` controla la apariencia visual en la interfaz. Los campos `createdAt` y `updatedAt` registran automáticamente las marcas de tiempo de creación y última modificación.

# User
User {
    idUser : INT [NOT NULL] [PRIMARY KEY]
    email : VARCHAR(255) [NOT NULL]
    password : VARCHAR(255) [NOT NULL]
}

**Descripción**:  
La entidad `User` representa a los usuarios del sistema. Cada usuario dispone de un identificador único (`idUser`), un correo electrónico (`email`) y una contraseña (`password`) para su autenticación.

# Permits
Permits {
    idDataset : INT [NOT NULL] [FOREIGN KEY -> Dataset.idDataset]
    idUser : INT [NOT NULL] [FOREIGN KEY -> User.idUser]
    isOwned : BOOLEAN [NOT NULL]
    [PRIMARY KEY (idDataset, idUser)]
}

**Descripción**:  
La entidad `Permits` modela la relación entre usuarios y datasets, permitiendo definir qué usuarios tienen acceso a qué datasets. Además, el atributo `isOwned` indica si el usuario es propietario del dataset o simplemente dispone de permisos sobre él.

# entry
entry {
    eid        : INT UNSIGNED [NOT NULL] [PRIMARY KEY]
    category   : VARCHAR(100) [NOT NULL]
    shape      : VARCHAR(50) [NULL]
    shape_type : VARCHAR(50) [NULL]
    size       : TINYINT UNSIGNED [NOT NULL]
}

**Descripción**:
La entidad `entry` representa una entrada individual del benchmark WebNLG. Cada entrada queda identificada por un código numérico sin signo (`eid`) y clasificada por categoría semántica (`category`). Los campos opcionales `shape` y `shape_type` describen la topología del grafo de triples, y `size` indica el número de triples que contiene la entrada.

# tripleset
tripleset {
    id   : INT UNSIGNED [NOT NULL] [PRIMARY KEY] [AUTO_INCREMENT]
    eid  : INT UNSIGNED [NOT NULL] [FOREIGN KEY -> entry.eid]
    type : ENUM('original','modified') [NOT NULL]
}

**Descripción**:
La entidad `tripleset` agrupa los triples RDF asociados a una entrada del benchmark. El atributo `type` distingue entre el conjunto de triples original, tal como aparece en el dataset fuente, y el conjunto modificado, que puede haber sido revisado o corregido. Una misma entrada puede tener varios triplesets de cada tipo.

# triple
triple {
    id           : INT UNSIGNED [NOT NULL] [PRIMARY KEY] [AUTO_INCREMENT]
    tripleset_id : INT UNSIGNED [NOT NULL] [FOREIGN KEY -> tripleset.id]
    subject      : VARCHAR(500) [NOT NULL]
    predicate    : VARCHAR(500) [NOT NULL]
    object       : VARCHAR(500) [NOT NULL]
}

**Descripción**:
La entidad `triple` almacena cada triple RDF individual perteneciente a un tripleset. Un triple queda definido por sus tres componentes: `subject` (sujeto), `predicate` (predicado) y `object` (objeto). Todos los valores se almacenan como cadenas de texto tal como aparecen en el benchmark.

# lex
lex {
    id      : INT UNSIGNED [NOT NULL] [PRIMARY KEY] [AUTO_INCREMENT]
    eid     : INT UNSIGNED [NOT NULL] [FOREIGN KEY -> entry.eid]
    lid     : VARCHAR(20) [NOT NULL]
    lang    : CHAR(5) [NOT NULL]
    comment : VARCHAR(500) [NULL]
    text    : TEXT [NOT NULL]
    [UNIQUE KEY (eid, lid, lang)]
}

**Descripción**:
La entidad `lex` almacena las frases de referencia (lexicalizaciones) asociadas a una entrada del benchmark. El campo `lid` agrupa las traducciones de una misma frase base (p. ej. `"Id1"`, `"Id2"`), y `lang` identifica el idioma en código ISO (p. ej. `"en"`, `"ru"`). El campo opcional `comment` permite anotar observaciones sobre la frase. La restricción de unicidad sobre `(eid, lid, lang)` garantiza que no existan duplicados para la misma combinación entrada–frase–idioma.

# dbpedialink
dbpedialink {
    id        : INT UNSIGNED [NOT NULL] [PRIMARY KEY] [AUTO_INCREMENT]
    eid       : INT UNSIGNED [NOT NULL] [FOREIGN KEY -> entry.eid]
    direction : VARCHAR(20) [NOT NULL]
    subject   : VARCHAR(500) [NOT NULL]
    predicate : VARCHAR(100) [NOT NULL]
    object    : VARCHAR(500) [NOT NULL]
}

**Descripción**:
La entidad `dbpedialink` almacena los enlaces entre entidades de DBpedia en distintos idiomas. El campo `direction` indica el sentido del enlace (p. ej. `"en2ru"`). El predicado es típicamente `"sameAs"`, relacionando la versión inglesa de una entidad con su equivalente en otro idioma. Estos enlaces provienen de la alineación multilingüe del dataset WebNLG.

# link
link {
    id        : INT UNSIGNED [NOT NULL] [PRIMARY KEY] [AUTO_INCREMENT]
    eid       : INT UNSIGNED [NOT NULL] [FOREIGN KEY -> entry.eid]
    direction : VARCHAR(20) [NOT NULL]
    subject   : VARCHAR(500) [NOT NULL]
    predicate : VARCHAR(100) [NOT NULL]
    object    : VARCHAR(500) [NOT NULL]
}

**Descripción**:
La entidad `link` almacena enlaces libres entre entidades que no figuran en DBpedia. Su estructura es análoga a `dbpedialink`, pero sirve para cubrir entidades propias del dominio que no disponen de entrada en DBpedia. Al igual que `dbpedialink`, se asocia siempre a una `entry` concreta mediante `eid`.

# Relaciones entre entidades

- Un **Dataset** puede tener múltiples **Section**.
- Una **Section** pertenece a un único **Dataset**.
- Un **User** puede tener permisos sobre múltiples **Dataset**.
- Un **Dataset** puede estar asociado a múltiples **User** mediante la tabla intermedia **Permits**.
- La entidad **Permits** resuelve la relación muchos-a-muchos entre **User** y **Dataset**.
- Una **entry** puede tener múltiples **tripleset** (tanto originales como modificados).
- Un **tripleset** pertenece a una única **entry** y contiene múltiples **triple**.
- Un **triple** pertenece a un único **tripleset**.
- Una **entry** puede tener múltiples **lex**, una por cada combinación de frase e idioma.
- Una **entry** puede tener múltiples **dbpedialink**.
- Una **entry** puede tener múltiples **link**.

## Restricciones del modelo

- `Section.idDataset` debe existir previamente en `Dataset.idDataset`.
- `Permits.idDataset` debe existir previamente en `Dataset.idDataset`.
- `Permits.idUser` debe existir previamente en `User.idUser`.
- La clave primaria compuesta de `Permits` garantiza que no pueda repetirse la misma relación usuario-dataset.
- Cada `Section` debe estar asociada obligatoriamente a un `Dataset`.
- `tripleset.eid` debe existir previamente en `entry.eid`. El borrado de una `entry` propaga la eliminación en cascada a todos sus `tripleset`.
- `triple.tripleset_id` debe existir previamente en `tripleset.id`. El borrado de un `tripleset` propaga la eliminación en cascada a todos sus `triple`.
- `lex.eid` debe existir previamente en `entry.eid`. El borrado de una `entry` propaga la eliminación en cascada a todas sus `lex`.
- La clave única `(eid, lid, lang)` de `lex` impide duplicados para la misma combinación de entrada, identificador de frase e idioma.
- `dbpedialink.eid` debe existir previamente en `entry.eid`. El borrado de una `entry` propaga la eliminación en cascada a todos sus `dbpedialink`.
- `link.eid` debe existir previamente en `entry.eid`. El borrado de una `entry` propaga la eliminación en cascada a todos sus `link`.

## Resumen del modelo

El modelo propuesto permite gestionar datasets de forma estructurada, dividiéndolos en secciones y controlando el acceso de los usuarios mediante permisos explícitos. Además, incorpora un subsistema completo para el almacenamiento de benchmarks en formato WebNLG: cada entrada del benchmark (`entry`) puede contener múltiples conjuntos de triples RDF —originales y modificados— así como frases de referencia multilingüe y enlaces de alineación entre entidades en DBpedia y fuentes externas. De este modo, la base de datos soporta tanto la organización interna del contenido y la gestión de permisos como el almacenamiento detallado de la información lingüística y semántica propia del benchmark.
